\section{Architectural Drivers}

The architecture was not chosen in a vacuum. The most important design drivers were ranked before the
service boundaries were fixed because different priorities would have produced a meaningfully different
system.

\begin{table}[H]
\centering
\caption{Ranked architectural drivers}
\label{tab:architectural-drivers}
\begin{tabularx}{\textwidth}{c p{4.2cm} X}
\toprule
\textbf{Rank} & \textbf{Driver} & \textbf{Why it mattered} \\
\midrule
1 & Collaboration correctness and convergence & The product fails if two users cannot trust what they are seeing while editing the same document. This pushed the design toward a CRDT-backed push model rather than naive polling or last-write-wins merging. \\
2 & Low-latency user experience & Typing, presence, and AI interaction feedback must feel responsive enough to resemble a real collaborative editor, not a batch workflow. \\
3 & Maintainable modular codebase & The assignment explicitly evaluates repository organization and feature decomposition. The architecture had to support multiple owners working in parallel without turning into a god-service monolith. \\
4 & AI safety, predictability, and cost control & AI is a core product feature, not a novelty. This drove the asynchronous job model, scoped prompting, and explicit policy/quota controls. \\
5 & Demo and reproducibility reliability & The MVP had to be runnable under grading pressure. This drove root orchestration scripts, SQLite portability, Docker Compose, CI, and deterministic reset behavior. \\
\bottomrule
\end{tabularx}
\end{table}

These rankings directly shaped the implementation: the realtime service is separated from the REST API,
AI uses asynchronous job handling rather than inline blocking calls, persistence is centralized in
SQLite for deterministic setup, and the repository is organized by service and domain boundaries rather
than by one large application file.

\section{AI Integration Design}

The AI assistant is treated as a core product feature, not a cosmetic API add-on. That means the design
must answer four questions simultaneously: what context the model sees, how users review and apply
results, how collaboration behaves while the AI is running, and how cost and failure are controlled.

\subsection{Context and Scope}

The delivered MVP sends the selected text plus a small amount of surrounding context to the AI provider.
This is the most pragmatic choice for coursework: it is cheaper than sending the full document, keeps
latency lower, and aligns with how rewrite/summarize/translate are actually invoked in the UI. The
backend contract requires the frontend to send a client-side selection snapshot containing the selected
range, the selected text, and the nearby context so that AI jobs can be linked to a concrete document
state.

For very long documents, the system intentionally does not default to sending the entire document. This
keeps token usage bounded and reduces privacy exposure. A future production design could add
chunked/restructure flows, but the delivered MVP focuses on scoped selection-based tasks that are easier
to reason about and test.

\subsection{Suggestion UX}

AI suggestions are presented in a side panel rather than silently modifying the document. This preserves
user control and matches the assignment's emphasis on AI as a product feature rather than a hidden
automation step. The delivered MVP supports:

\begin{itemize}
  \item full apply of the AI output,
  \item partial apply by selecting only a portion of the AI output,
  \item explicit rejection without modifying document content,
  \item immediate visibility of AI job state.
\end{itemize}

Applying AI is treated as a document-changing action and therefore creates version snapshots for
traceability. This design also aligns with the version history and revert model: AI is not special-cased
 into an untracked side path.

\subsection{AI During Collaboration}

The design intentionally avoids hard-locking a region when AI is invoked. Instead, the system snapshots
the selected text, lets the AI run asynchronously, and allows other collaborators to continue editing.
When the AI response arrives, the frontend presents it as a proposal. If the relevant text has changed in
the meantime, the user must explicitly decide whether to apply or discard the AI output. This is a
reconcile-first design rather than a lock-first design.

This choice is justified by the architectural drivers. Locking would be simpler to explain, but it would
damage collaboration fluidity and create awkward waiting states. Reconciliation with explicit user review
keeps the collaboration model more honest, even if it requires careful versioning and conflict-safe save
behavior.

\subsection{Prompt Design and Provider Strategy}

Prompts are task-specific and template-based. The frontend chooses the user-facing action
(rewrite/summarize/translate), while the backend/provider layer is responsible for constructing the final
provider request. This keeps prompt construction out of the browser and allows prompt logic or provider
selection to evolve without changing the frontend contract.

The delivered MVP uses a provider abstraction with two concrete modes:

\begin{itemize}
  \item \texttt{stub}: a deterministic provider used for reliable demos and tests,
  \item \texttt{lmstudio}: a local OpenAI-compatible provider used when a real local model endpoint is
  available.
\end{itemize}

This is a pragmatic compromise between the architectural goal of AI usefulness and the assignment's need
for repeatable demonstrations.

\subsection{Async Job Model}

AI operations are modeled as asynchronous jobs. The backend validates access, policy, quota, and input
shape, creates a job record, and returns quickly with \texttt{jobId}, \texttt{statusUrl}, and an initial
status. The frontend polls until the job reaches \texttt{SUCCEEDED} or \texttt{FAILED}. This model avoids
blocking request threads and gives the client a precise way to distinguish “still running” from “failed”
and “succeeded.”

\section{API Design}

The API style follows a hybrid model:

\begin{itemize}
  \item \textbf{REST} for CRUD, permissions, versions, exports, AI jobs, and session issuance,
  \item \textbf{WebSocket + Yjs} for low-latency collaborative state propagation,
  \item \textbf{Polling} from the frontend only for long-running AI jobs and export jobs.
\end{itemize}

This division is deliberate. REST keeps persistence-oriented operations predictable and easy to test.
Realtime collaboration is handled by a dedicated push channel because document editing is inherently a
shared live interaction. Long-running AI and export operations fit naturally into job-based REST flows.

\subsection{Core API Contracts}

The assignment brief requires at least document CRUD, realtime session management, AI invocation, and
user/permission management. The delivered backend provides these contracts concretely and documents them
through OpenAPI and contract notes in the repository.

All non-2xx responses use a shared JSON error shape:

\begin{quote}
\texttt{\{ "error": \{ "code": "<CODE>", "message": "<MESSAGE>", "details": \{ ...optional... \} \} \}}
\end{quote}

This is important because the assignment brief explicitly asks how the client can distinguish between AI
latency, AI failure, quota issues, permission problems, and not-found conditions. The standard error
schema gives the frontend a uniform contract for all of those cases.

\subsection{Communication Model}

The communication model is intentionally hybrid.

\begin{itemize}
  \item \textbf{Document open flow.} The frontend first loads a document snapshot through REST and then
  requests a signed realtime session URL from the backend.
  \item \textbf{Live collaboration.} The realtime service validates the signed session token and relays Yjs
  sync and awareness updates among collaborators.
  \item \textbf{Disconnect behavior.} If realtime becomes unavailable, the frontend falls back to offline/manual-save mode.
  \item \textbf{AI completion.} AI uses async jobs with polling instead of websocket completion events in
  the MVP because the job semantics are easier to reason about and test.
\end{itemize}

\section{Code Structure and Repository Organization}

The final repository intentionally uses a monorepo. For a four-person academic team with strongly
connected frontend, backend, realtime, and AI layers, a monorepo reduces coordination overhead and makes
contract drift easier to detect.

\subsection{Top-Level Layout}

\begin{lstlisting}[basicstyle=\ttfamily\small]
collaborative-editor-ai/
+-- frontend/        React + TypeScript client
+-- backend/         Express API, SQLite persistence, RBAC, AI jobs, export
+-- realtime/        WebSocket + Yjs relay service
+-- ai-service/      AI provider abstraction and prompt execution
+-- shared/          shared contracts such as session token signing
+-- docs/            architecture notes, Mermaid sources, diagram exports
+-- scripts/         root install/dev/test/demo orchestration
+-- tests/           root integration smoke tests
`-- final-report/    final written submission package
\end{lstlisting}

\subsection{Shared Code and Config}

The shared layer is intentionally narrow. It contains only cross-service helpers that justify shared
ownership, such as session-token signing/verification. This avoids over-abstracting everything into a
shared package while still keeping critical contracts consistent. Configuration is managed through
environment variables and \texttt{.env.example} files; secrets are excluded from version control.

\subsection{Testing Structure}

The repository uses layered testing rather than relying on only one type of test.

\begin{itemize}
  \item Backend: unit/integration tests with Node's built-in test runner
  \item Frontend: component/integration-style tests with Vitest
  \item Realtime: websocket/Yjs integration tests with Node's built-in test runner
  \item Root: a smoke integration test that starts backend and realtime together and validates wiring
\end{itemize}

This structure mirrors the feature decomposition: REST, realtime, and UI are each tested in their own
layer, and the root smoke test catches cross-service drift.

\section{Architecture Decision Records}

\subsection*{ADR-001 --- Realtime consistency strategy}
\textbf{Status:} Accepted.\\
\textbf{Context:} The system needs simultaneous editing, reconnect safety, and convergence without the
team implementing a custom operational transform algorithm.\\
\textbf{Decision:} Use a CRDT-based collaboration approach built on Yjs, with the realtime service acting
as a relay and permission gate rather than a custom transform engine.\\
\textbf{Consequences:} This improves convergence and offline/reconnect resilience, but requires careful
handling of room lifecycle, awareness state, and persistence boundaries.\\
\textbf{Alternatives considered:} Custom OT server logic, naive last-write-wins, and polling-based text
sync were rejected due to complexity or poor correctness.

\subsection*{ADR-002 --- API style and async job model}
\textbf{Status:} Accepted.\\
\textbf{Context:} The system mixes CRUD-like document operations with long-running AI and export actions.\\
\textbf{Decision:} Use REST for document, permission, version, AI job, export job, and session endpoints;
use WebSocket for realtime collaboration; and model AI/export as asynchronous jobs.\\
\textbf{Consequences:} The API is concrete and testable, but the frontend must poll job status and the
backend must persist job state.\\
\textbf{Alternatives considered:} synchronous AI calls, GraphQL for all interactions, and polling for live
collaboration were rejected.

\subsection*{ADR-003 --- Repository structure strategy}
\textbf{Status:} Accepted.\\
\textbf{Context:} The team needed a code structure that matched a multi-service system while minimizing
coordination overhead for a small academic group.\\
\textbf{Decision:} Use a monorepo with service directories and keep persistence logic grouped by domain
repositories rather than a single giant application repository file.\\
\textbf{Consequences:} Shared contracts are easier to keep aligned, but discipline is required to prevent
tight coupling.\\
\textbf{Alternatives considered:} separate repositories and hybrid split-repo models were rejected due to
contract drift and coordination cost.

\subsection*{ADR-004 --- AI provider strategy}
\textbf{Status:} Accepted.\\
\textbf{Context:} The system needs rewrite, summarize, and translate behaviors that are useful during a
demo but not dependent on an always-available external cloud provider.\\
\textbf{Decision:} Keep AI behind a provider abstraction supporting a deterministic stub mode and a local
LM Studio/OpenAI-compatible mode.\\
\textbf{Consequences:} Demo reliability improves and tests remain deterministic, but the system is still
architected so a different provider can be added later.\\
\textbf{Alternatives considered:} hard-wiring one hosted provider or requiring self-hosted production-scale
model infrastructure for the MVP.
